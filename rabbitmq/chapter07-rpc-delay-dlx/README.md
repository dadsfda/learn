# 第七章：RPC、延迟与死信

> RabbitMQ 可以做请求响应、延迟任务和失败兜底，但这些能力要谨慎使用。

## 目标

- 理解 RPC 模式
- 理解 TTL + 死信交换机实现延迟任务
- 理解失败消息为什么需要死信队列

## 1. RPC：通过队列做请求响应

普通消息是单向的：

```text
Producer -> Queue -> Consumer
```

RPC 是请求响应：

```text
Client -> request queue -> Worker
Client <- reply queue <- Worker
```

请求消息通常带两个字段：

| 字段 | 作用 |
|------|------|
| `reply_to` | 告诉服务端结果发到哪个队列 |
| `correlation_id` | 匹配请求和响应 |

适合：

- 后台服务之间需要异步解耦，但又要拿到结果
- 处理时间不稳定，不想直接 HTTP 等待

不适合：

- 简单同步查询
- 高频低延迟接口
- 能直接 HTTP/gRPC 调用的简单场景

## 2. 延迟任务

RabbitMQ 常见延迟任务方案：TTL + 死信交换机。

流程：

```text
消息进入延迟队列
-> 等待 TTL 过期
-> 消息变成死信
-> 被投递到死信交换机
-> 路由到真正消费队列
```

队列参数示例：

```go
args := amqp.Table{
    "x-message-ttl":             int32(60000),
    "x-dead-letter-exchange":    "delay.dlx",
    "x-dead-letter-routing-key": "order.timeout",
}
```

含义：

- 消息在延迟队列等待 60 秒。
- 过期后投递到 `delay.dlx`。
- 使用 routing key `order.timeout`。

## 3. 实际应用：订单超时取消

业务流程：

```text
用户创建订单
-> 写订单表，状态为待支付
-> 发送延迟消息，30 分钟后触发
-> 消费者收到 order.timeout
-> 查询订单状态
-> 如果仍未支付，取消订单并释放库存
-> 如果已支付，忽略消息
```

关键点：消费者必须查数据库确认当前状态，不能只相信延迟消息。

## 4. 死信队列 DLQ

消息会成为死信的常见原因：

- 消息被 `Nack(false, false)` 拒绝
- 消息过期
- 队列长度超过限制

死信队列的作用：

- 保存失败消息，方便排查
- 避免失败消息无限重试拖垮系统
- 支持人工修复后重新投递

## 5. 重试建议

不要无限 `Nack(false, true)`。

更稳的方式：

```text
第一次失败 -> 短延迟重试
第二次失败 -> 长一点延迟重试
第三次失败 -> 进入死信队列
人工或补偿任务处理
```

消息体里可以带重试次数：

```json
{
  "event_id": "evt_1001",
  "retry_count": 0,
  "payload": {
    "order_id": 1001
  }
}
```

## 6. RPC 使用提醒

RPC 容易让系统重新变成“同步等待”，用之前先问：

1. 是否真的需要立即拿到结果？
2. 直接 HTTP/gRPC 是否更简单？
3. 超时后调用方怎么处理？
4. 响应消息丢了怎么办？

大多数业务事件不需要 RPC，普通异步消息就够用。
