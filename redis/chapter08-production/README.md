# 第八章：生产实践

> Redis 很快，但不是无脑快。生产环境要关注命名、内存、持久化和故障风险。

## 目标

- 掌握实用的 key 命名规范
- 理解内存和过期策略
- 了解持久化、监控和危险命令

## 1. key 命名规范

推荐格式：

```text
业务:对象:id:字段
```

示例：

```text
user:1001:profile
product:1001:detail
article:1001:liked_users
rank:game:daily:2026-06-06
rate:login:ip:127.0.0.1
```

建议：

- 使用英文小写。
- 用冒号分隔层级。
- key 不要太短到看不懂。
- key 也不要太长，避免浪费内存。
- 临时数据必须设置过期时间。

## 2. value 不要太大

Redis 是内存数据库，大 value 会带来问题：

- 网络传输慢
- 序列化和反序列化慢
- 删除时可能阻塞
- 内存占用高

经验建议：

- 缓存接口结果时，只缓存真正需要的字段。
- 不要把超大列表整块塞进一个 String。
- 大集合要考虑拆分或者换数据库方案。

## 3. 过期时间设计

常见 TTL：

| 数据 | 建议过期时间 |
|------|--------------|
| 验证码 | 5 分钟 |
| 登录 token | 2 小时到 7 天，按业务决定 |
| 商品详情缓存 | 5 到 30 分钟 |
| 热点榜单 | 1 到 10 分钟 |
| 限流 key | 1 到 60 分钟 |

缓存类 key 建议加随机过期时间，避免同一时间大量失效：

```text
实际 TTL = 基础 TTL + 随机 0 到 120 秒
```

## 4. 持久化基础

Redis 常见持久化方式：

| 方式 | 说明 |
|------|------|
| RDB | 定期生成快照，恢复快，可能丢失最近一段数据 |
| AOF | 记录写命令，数据更完整，文件通常更大 |

小白阶段先记住：

- Redis 做缓存时，即使数据丢了也能从数据库重建。
- Redis 保存登录态、队列、计数等数据时，要考虑重启后丢失的影响。
- 关键业务不要只依赖单机 Redis 内存数据。

## 5. 内存淘汰策略

当 Redis 内存满了，会根据配置决定怎么处理。

常见策略：

| 策略 | 含义 |
|------|------|
| `noeviction` | 内存满后写入报错 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 |
| `volatile-lru` | 只从设置了过期时间的 key 中淘汰最近最少使用的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |

实际项目里，缓存型 Redis 常用 `allkeys-lru`；混合存储时要更谨慎。

## 6. 谨慎使用危险命令

生产环境避免使用：

```bash
KEYS *
FLUSHDB
FLUSHALL
```

原因：

- `KEYS *` 数据多时可能阻塞 Redis。
- `FLUSHDB` 会清空当前库。
- `FLUSHALL` 会清空所有库。

需要查找 key 时，优先使用 `SCAN`：

```bash
SCAN 0 MATCH product:* COUNT 100
```

`SCAN` 是渐进式扫描，不会一次性遍历全部 key。

## 7. 监控重点

生产环境至少关注：

| 指标 | 为什么重要 |
|------|------------|
| 内存使用量 | Redis 主要吃内存 |
| key 数量 | 判断是否有 key 泄漏 |
| 命中率 | 缓存是否真的有效 |
| 慢查询 | 是否有阻塞命令 |
| 连接数 | 是否连接泄漏 |
| 持久化状态 | 是否保存失败 |

常用命令：

```bash
INFO memory
INFO stats
INFO clients
SLOWLOG GET 10
DBSIZE
```

## 8. 实际应用检查清单

上线一个 Redis 功能前，问自己：

1. key 名能看懂吗？
2. 是否需要过期时间？
3. value 会不会太大？
4. 数据丢失后能不能恢复？
5. 缓存是否会和数据库不一致？
6. 热点 key 过期时会不会打爆数据库？
7. 是否用了 `KEYS *` 这类危险命令？
8. 是否需要限流、锁或重试保护？

## 练习

1. 为商品详情、登录 token、排行榜分别设计 key。
2. 给不同业务数据设计 TTL。
3. 使用 `INFO memory` 查看 Redis 内存信息。
4. 使用 `SCAN` 查找 `product:*` 类型 key。
